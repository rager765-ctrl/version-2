cached 2. Stale-While-Revalidate Caching Model

How the storefront loads catalog data instantly (<1ms) using local cache first, then asynchronously updates in the background, falling back to direct Firestore snapshot listeners if Render is offline. 

sequenceDiagram
    autonumber
    actor User as Browser UI / Client
    participant Store as store.js (Client Store)
    participant Disk as Local Disk (LocalStorage)
    participant Backend as Node.js Server (Render)
    participant Firebase as Cloud Firestore

    User->>Store: Initialize Storefront Page
    Store->>Disk: Request cached catalog datasets
    Disk-->>Store: Return Cached JSON (Instant <1ms)
    Store-->>User: Render Catalog UI (Zero Flash)

    Store->>Backend: REST fetch /api/products (Validate)
    alt Node.js Server Online
        Backend-->>Store: Return latest data (Write & Stamp TTL)
        Store-->>User: Silently update UI
    else Node.js Server Offline / Sleeping
        Store-->>Store: Server Timeout (60s reached)
        Store->>Firebase: Activate direct fallback listener
        Firebase-->>Store: Stream live data changes
        Store->>Disk: Write fresh data to Disk Cache
        Store-->>User: Update UI state smoothly
    end

Disk Storage Quota Shield

If localStorage hits its size limit (QuotaExceededError), our custom storage system automatically purges large, stale general lists while protecting and preserving the user's active shopping cart and credentials.

Zero Flash Page Boots

Serving the entire catalog and UI configurations synchronously in under 1ms eliminates skeletal flashing entirely, keeping layout transitions extremely premium.

payments 5. Billing Optimization & Cost-Savings Formula

Mathematical breakdown demonstrating how reads multiplexing through the Node.js backend reduces Firestore monthly read bills down to practically zero.

🚀 Native Direct Reads/Day = N × (Products + Categories + Sellers + Orders)
🛡️ Node.js Capped Reads/Day = 1 × (Products + Categories + Sellers + Orders)

High-Traffic Scenario Simulation

For **1,500 daily campus visitors** browsing a catalog of **175 total items**:

❌ **Direct Firebase Sync:** 1,500 × 175 = **262,500 billable reads/day** (Bypasses free Spark Tier limits instantly!)

✅ **Optimized Caching Server:** 1 × 175 = **175 billable reads/day** (Uses only **0.35%** of Spark Tier daily quota, keeping operations **100% FREE**).
